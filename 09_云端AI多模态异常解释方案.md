# 09 云端 AI 多模态异常解释方案

## 9.1 AI 层定位

云端 AI 智能解释层是本项目四层架构的最上层。它的职责是对异常事件的抓拍图片和传感器数据进行智能分析，输出异常类别、场景描述、风险评估和处理建议。

核心原则：
- AI 结果只用于解释、展示、归档和辅助判断。
- AI 不直接控制继电器、水泵、蜂鸣器等关键执行器。
- STM32 本地危险判断优先级最高。
- 如果 AI 请求失败，Web/手机端显示"AI 分析不可用"，本地系统继续运行。

---

## 9.2 AI 调用流程

```text
STM32 触发报警 → ESP32-CAM 抓拍成功
→ ESP32-CAM 上传事件 JSON + 图片至服务器
→ 服务器收到后自动（或手动触发）调用云端 AI
→ 服务器将图片 + 传感器上下文组织为 AI 请求 JSON
→ 调用云端视觉模型 API 或大模型 API
→ AI 返回异常解释 JSON
→ 服务器存储 AI 结果，关联到对应事件
→ Web/手机端刷新时展示 AI 解释结果
→ 可选：服务器通过 MQTT 或 ESP32-CAM 回传 AI 摘要给 STM32 OLED 显示

AI 失败时：
→ 服务器标记 ai_status = "unavailable"
→ Web 页面显示"AI 分析不可用，请查看传感器数据和抓拍图片"
→ 本地报警和执行器动作不受影响
```

---

```json
{
  "schema_ver": "2.0",
  "request_id": "REQ_20260601_103000_E000012",
  "event_id": "E000012",
  "timestamp": "2026-06-01T10:30:00+08:00",
  "device_id": "labbox_001",
  "risk_level_local": "alarm",
  "risk_score": 7,
  "event_type": "SMOKE_ALARM",
  "sensor_snapshot": {
    "smoke_adc": 1850,
    "smoke_warn": true,
    "smoke_alarm": true,
    "flame": false,
    "pir": true,
    "door_open": false,
    "temp_c": 46.2,
    "humi_pct": 55.0,
    "vib_rms_g": 0.12,
    "vib_pp_g": 0.30
  },
  "image_url": "http://192.168.1.100:5000/uploads/E000012.jpg",
  "image_base64": "<可选，图片 base64 编码>",
  "local_action": "FAN_ON,BUZZER_FAST,RGB_RED",
  "system_state": "alarm",
  "context": "实验室设备柜安全巡检系统，STM32本地检测到烟雾异常，已触发风扇和声光报警"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| event_id | string | 是 | 事件唯一标识 |
| timestamp | string | 是 | 事件发生时间 |
| device_id | string | 是 | 设备编号 |
| risk_level_local | string | 是 | STM32 本地判断的风险等级 |
| risk_score | int | 是 | 本地风险分数 |
| event_type | string | 是 | 事件类型 |
| sensor_snapshot | object | 是 | 传感器数据快照 |
| image_url 或 image_base64 | string | 至少一个 | 抓拍图片 |
| local_action | string | 是 | 本地已执行的动作 |
| system_state | string | 是 | 当前系统状态 |
| context | string | 推荐 | 提供给 AI 的场景描述上下文 |

## 9.4 AI 返回 JSON 格式

```json
{
  "schema_ver": "2.0",
  "event_id": "E000012",
  "ai_status": "ok",
  "ai_detected_type": "possible_smoke",
  "ai_risk_level": "warning",
  "scene_description": "图像中疑似存在轻微烟雾或雾气，设备柜内部可见模糊区域。结合传感器数据显示 MQ-2 读数偏高（1850），温度 46.2°C，综合判断存在轻度烟雾异常可能。",
  "reason": "MQ-2 传感器读数超过预警阈值，图像中可见疑似烟雾区域，温度偏高但未达危险水平",
  "suggestion": "建议：1. 检查设备柜内部是否有过热元件或电线接触不良；2. 保持通风，持续监测烟雾浓度变化；3. 如烟雾持续增强，应立即断电并检查",
  "confidence": 0.82,
  "should_notify": true,
  "display_text": "【AI 分析】疑似轻微烟雾，建议检查设备柜内部过热元件",
  "version": "gpt-4o-mini"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| ai_status | string | 是 | "ok" / "error" / "timeout" |
| ai_detected_type | string | 是 | 异常类别：normal / possible_smoke / fire / intrusion / vibration / unknown |
| ai_risk_level | string | 是 | AI 判断的风险等级：normal / warning / alarm / danger |
| scene_description | string | 是 | 中文场景描述 |
| reason | string | 是 | 异常原因分析 |
| suggestion | string | 是 | 处理建议 |
| confidence | float | 是 | 置信度 0.0-1.0 |
| should_notify | bool | 推荐 | 是否建议推送通知 |
| display_text | string | 推荐 | 适合在有限屏幕上显示的一行文本 |
| version | string | 推荐 | 使用的 AI 模型版本 |

## 9.5 AI 调用实现方式

### 方式一：调用云端大模型 API（推荐）

使用 OpenAI GPT-4o-mini / Claude API / 通义千问 等视觉大模型 API。

服务器侧 Python 伪代码：

```python
import requests, json, base64

def analyze_event(event_json, image_path):
    with open(image_path, "rb") as f:
        img_b64 = base64.b64encode(f.read()).decode()

    prompt = f"""你是一个安全巡检 AI 助手。请分析以下异常事件：
事件类型：{event_json['event_type']}
本地风险等级：{event_json['risk_level_local']}
传感器数据：{json.dumps(event_json['sensor_snapshot'], ensure_ascii=False)}
本地已执行动作：{event_json['local_action']}

请根据图片和传感器数据，返回 JSON 格式分析结果：
{{"ai_detected_type": "...", "ai_risk_level": "...",
  "scene_description": "...", "reason": "...",
  "suggestion": "...", "confidence": 0.0-1.0,
  "display_text": "一行摘要"}}
仅返回 JSON，不要其他文字。"""

    response = requests.post(
        "https://api.openai.com/v1/chat/completions",
        headers={"Authorization": f"Bearer {API_KEY}"},
        json={
            "model": "gpt-4o-mini",
            "messages": [{"role": "user", "content": [
                {"type": "text", "text": prompt},
                {"type": "image_url", "image_url": {
                    "url": f"data:image/jpeg;base64,{img_b64}"
                }}
            ]}],
            "max_tokens": 500
        },
        timeout=15
    )

    result = response.json()
    ai_text = result["choices"][0]["message"]["content"]
    return json.loads(ai_text.strip())
```

### 方式二：使用模拟/预设 AI 返回（保底）

如果云端 API 不可用或 API Key 不方便使用，可以在服务器侧根据传感器数据生成规则化的"模拟 AI 返回"。

```python
def mock_ai_analyze(event_json):
    event_type = event_json.get("event_type", "UNKNOWN")
    rules = {
        "SMOKE_ALARM": {"type": "possible_smoke", "level": "warning",
                        "desc": "传感器检测到烟雾异常，建议检查通风"},
        "FLAME":       {"type": "fire", "level": "danger",
                        "desc": "检测到火焰信号，建议立即断电检查"},
        "DOOR_OPEN":   {"type": "intrusion", "level": "alarm",
                        "desc": "检测到柜门被打开，可能存在非法入侵"},
        "VIBRATION":   {"type": "vibration", "level": "warning",
                        "desc": "检测到异常震动，可能存在设备被移动"},
    }
    r = rules.get(event_type, {"type": "unknown", "level": "warning",
                                "desc": "检测到未知异常"})
    return {
        "ai_status": "ok",
        "ai_detected_type": r["type"],
        "ai_risk_level": r["level"],
        "scene_description": r["desc"],
        "reason": f"基于{event_type}事件和传感器数据的规则化分析",
        "suggestion": "建议检查设备柜并持续监测",
        "confidence": 0.75,
        "display_text": f"【分析】{r['desc']}",
        "version": "rule_based_v1"
    }
```

答辩时可以说明：项目支持接入真实云端 AI API，也内置了规则化分析引擎作为 AI 不可用时的降级方案。

## 9.6 AI 错误处理

| 错误场景 | 处理方式 |
|---|---|
| AI API 超时（>15 秒） | 标记 ai_status = "timeout"，Web 显示"AI 分析超时" |
| AI API 返回非 JSON | 标记 ai_status = "error"，记录原始返回 |
| AI API 返回无效字段 | 使用默认值填充缺失字段 |
| API Key 无效 | 标记 ai_status = "auth_error"，使用模拟返回 |
| 网络不可达 | 标记 ai_status = "network_error"，使用模拟返回或跳过 |
| 图片过大 | 压缩至 640x480 后再发送 |

所有 AI 错误均不影响 STM32 本地安全判断和执行器动作。

## 9.8 AI 结果回传 STM32（可选）

如果实现了 AI_RESULT UART 命令，ESP32-CAM 或服务器可以将 AI 摘要回传给 STM32。

STM32 收到后可在 OLED 显示：

```text
AI: possible_smoke
CONF: 82%
TIP: check overheat
```

STM32 对 AI 结果的处理规则：
- AI 结果只显示在 OLED 和串口日志。
- AI 结果不改变本地风险评分。
- AI 结果不控制任何执行器。
- AI 超时/失败时 OLED 显示 "AI: N/A"。

## 9.9 AI 可识别的异常类别

| 类别 | 英文标签 | 说明 |
|---|---|---|
| 正常 | normal | 图像和传感器数据无异常 |
| 疑似烟雾 | possible_smoke | 图像中可见烟雾/雾气 |
| 火焰/火灾 | fire | 图像中可见明火或强光 |
| 入侵/闯入 | intrusion | 图像中可见非预期人员或物体 |
| 设备异常 | equipment_fault | 图像中可见倒塌/散落/位移 |
| 异常震动 | vibration | 结合传感器数据判断 |
| 未知异常 | unknown | 无法归类的异常 |

## 9.10 为什么采用"AI 辅助解释"而非"AI 直接控制"

| 设计选择 | 原因 |
|---|---|
| AI 不直接控制水泵 | AI 推理延迟 1-15 秒，安全保护需要毫秒级响应 |
| AI 不覆盖本地判断 | 网络中断时 AI 不可用，安全系统不能停摆 |
| AI 不降低风险等级 | 避免 AI 误判导致安全保护被取消 |
| AI 结果用于人类决策 | 帮助运维人员理解异常原因和制定后续措施 |
| 本地传感器优先 | 传感器在毫秒级持续工作，比网络依赖的 AI 更可靠 |

这是安全关键系统的工程最佳实践：**端侧保障实时性和可靠性，云端提供智能化和可解释性**。
