# 平台迁移决策记录

> 日期：2026-06-04
> 结论：以 i.MX6ULL Pro + Orange Pi 5 为新主线，STM32/ESP32-CAM 归档为 legacy。

## 1. 迁移背景

原项目采用 STM32 + ESP32-CAM + Flask + 云端 AI。随着调试推进，硬件接线和单片机/ESP32-CAM 串口桥接消耗较多时间，而项目目标逐渐转向本地 AI 与高效开发。因此决定将第一层、第二层合并迁移到 i.MX6ULL Pro Linux，将第四层改为 Orange Pi 5 本地 RKNN 推理。

## 2. 原 STM32/ESP32 已验证资产

| 资产 | 状态 | 新项目处理 |
|---|---|---|
| STM32 BSP 分层 | 有参考价值 | 归档，思想用于 Linux 模块划分 |
| `app_comm` JSON | 有参考价值 | 字段语义迁移到 HTTP Contract |
| ESP32-CAM WiFi POST | 历史已验证 | 归档，HTTP 思路可参考 |
| Flask `/api/events` | 当前可复用 | 保留并扩展 |
| Dashboard | 当前可复用 | 保留并扩展 |
| 测试记录 | 可作为工程过程证据 | 放入 `tests/legacy/` |

## 3. 原链路阻塞点

最新会话状态显示：ESP32-CAM 网络侧可 POST 到 Flask，STM32 PA9/USART1_TX 可输出 115200 8N1 JSON；但 STM32 PA9 → ESP32-CAM GPIO13 → Flask → Dashboard 未最终验收。当前 zip 中的代码仍是 USART2/huart2 与 ESP32-CAM UART0/GPIO3 方案，已经落后于最新排查状态。

处理原则：不继续修 PA9/GPIO13 物理桥接，不把 Task04 写成通过，只归档为历史状态。

## 4. 新平台选择理由

| 选择 | 理由 |
|---|---|
| i.MX6ULL Pro | 嵌入式 Linux、SSH 部署、GPIO/I2C/PWM/V4L2 可覆盖底层数字系统内容 |
| Orange Pi 5 | RK3588S NPU、本地 RKNN 推理、NVMe 存储、适合本地 AI 服务 |
| Flask 保留 | 已有接口和 Dashboard，是最成熟资产 |
| HTTP JPEG/JSON | curl 可测、调试直观、两周周期最稳 |

## 5. 资产去留表

| 资产 | 判断 | 概念可复用 | 实现需重写 |
|---|---|---|---|
| `firmware/stm32/` | 归档 | 状态机、BSP 分层、JSON 字段 | HAL、Keil、CubeMX 不适用 i.MX6ULL |
| `firmware/esp32cam/` | 归档 | WiFi POST、HTTP 超时思路 | ESP32-CAM 不再承担抓拍/联网 |
| `server/backend/` | 保留更新 | Flask、SQLite、Dashboard、图片上传 | 新增 vision/AI/image 字段和 UI |
| docs/06 | 更新重写 | risk_score、状态机 | C/HAL 实现改 Linux 规则引擎 |
| docs/07 | 重写 | JSON 语义 | UART 主从变 HTTP Contract |
| docs/09 | 重写 | AI 解释/失败降级 | 云端 API 变 OPi5 RKNN |

## 6. 风险与回滚

| 风险 | 回滚 |
|---|---|
| i.MX SDK 配置慢 | 先板端原生编译或 Python 骨架 |
| RKNN 接入慢 | mock AI 服务完成 Contract 与 Dashboard |
| 摄像头抓帧慢 | 降分辨率或使用测试图片 |
| 云台供电不稳 | 改固定摄像头，不影响主链路 |
| 答辩质疑实时性 | 强调人机尺度、安全默认态、watchdog、离线降级 |

## 7. 对答辩叙事的影响

迁移不是失败，而是工程决策：第一版验证了 Flask 与事件 JSON 方向，暴露了单片机/ESP32 物理调试成本；第二版利用嵌入式 Linux 和本地 NPU 提升开发效率和 AI 展示能力，同时保留 GPIO/I2C/PWM/MOS 等课程关键点。
