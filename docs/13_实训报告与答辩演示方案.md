## 13.2 报告中的 AI/Web 相关章节写作要点

### 第 9 章 Web Dashboard 与服务器方案 写作建议

重点展示以下内容：

1. Web Dashboard 页面设计（状态卡片、传感器面板、事件列表、抓拍图片、AI 解释结果）。
2. REST API 设计表（至少 8 个端点：状态查询、事件列表、图片获取、AI 结果查询等）。
3. 服务器部署方案（Flask 单文件即可，不需要复杂框架）。
4. 手机端演示截图（手机浏览器访问局域网 IP 即可，不需要原生 APP）。
5. 配上 Web 页面 Mockup 截图或实际运行截图。

报告话术示例：

> 远程管理层采用 Flask 轻量后端，提供 RESTful API 接收 ESP32-CAM 上传的事件数据和抓拍图片，并通过 Web Dashboard 向 PC 和手机端提供实时状态可视化、历史事件查询和 AI 解释结果展示。手机端通过浏览器访问局域网地址即可查看巡检状态，无需安装原生 APP。

### 第 10 章 云端 AI 多模态异常解释 写作建议

重点展示以下内容：

1. AI 请求 JSON 完整结构（含 sensor_snapshot、image_url/base64、context）。
2. AI 响应 JSON 完整结构（含 ai_detected_type、scene_description、reason、suggestion、confidence）。
3. Python 调用代码（真实 API 版 + Mock 降级版，建议都写入报告）。
4. AI 结果在 Web 端的展示效果（AI 可用 vs AI 不可用两种界面状态）。
5. "AI 辅助解释 vs AI 直接控制"设计决策说明。

报告话术示例：

> 云端 AI 智能解释层将异常事件的传感器快照和抓拍图片组织为结构化 JSON，通过 HTTPS 调用云端视觉大模型 API，获取异常类别识别、场景描述、原因分析和处置建议。AI 分析结果以结构化 JSON 返回，在 Web Dashboard 中以卡片形式展示。当 AI 服务不可用时，系统自动降级为"AI 分析不可用——本地判断生效"，确保本地安全闭环不受影响。

## 13.6 3-5 分钟答辩演示流程（A 类/AI 导向版）

### 0:00-0:30 开场介绍

讲稿：

> 老师好，我们的项目是基于 STM32F103C8T6 的端云协同 AI 多模态安全巡检与设备健康监测系统。系统采用四层架构：STM32 本地实时控制层负责多传感器采集、风险评分、状态机和执行器联动；ESP32-CAM 边缘视觉层完成异常现场抓拍和取证；Flask 服务器与 Web Dashboard 提供远程状态可视化和手机端查看；云端 AI 对异常图片进行多模态解释和处置建议生成。整个系统即使摄像头、网络、AI 全部失效，STM32 本地安全闭环仍可独立工作。

### 0:30-1:10 正常状态与布防

操作：

1. 上电，OLED 显示启动信息和自检结果。
2. 按 K1 布防。
3. ARM_DELAY 倒计时，RGB 黄闪，蜂鸣器短响。
4. 进入 NORMAL，RGB 绿色常亮，蜂鸣器关闭。
5. 打开手机浏览器，访问 Web Dashboard，展示正常状态页面。

讲解：

> 系统上电后完成外设自检，按布防键进入监控状态。Web Dashboard 和手机端同步显示当前传感器数值和系统状态。

### 1:10-1:50 传感器异常与本地报警

操作：

1. 触发 MQ-2 或调阈值模拟烟雾。
2. 打开门磁或触发 PIR。
3. 观察风险分数从正常逐步升高。
4. 系统进入 PRE_ALARM → ALARM。
5. 蜂鸣器快响，RGB 红闪，风扇启动。

讲解：

> STM32 根据 6 种传感器数据进行多模态风险评分。风险等级从正常、预警、报警到危险分级响应。所有判断和执行器控制由 STM32 本地完成，不依赖网络。

### 1:50-2:30 ESP32-CAM 抓拍与 Web 可视化

操作：

1. 报警触发后，OLED 显示 SNAP_REQ → ACK → SNAP_OK。
2. 切换到手机/电脑 Web Dashboard，展示最近抓拍图片和事件列表。
3. 展示事件详情页：传感器快照、抓拍图片、风险等级。

讲解：

> 异常发生时 STM32 通过 UART 调度 ESP32-CAM 抓拍取证。协议支持 ACK 确认、心跳检测和超时重传。抓拍图片和传感器快照通过服务器保存，可在 Web 端和手机端远程查看。

### 2:30-3:10 云端 AI 异常解释

操作：

1. 在 Web Dashboard 上展示 AI 解释结果卡片。
2. 展示内容：异常类别、场景描述、原因分析、处置建议、置信度。
3. 可以使用真实 API 返回或预录截图。

讲解：

> 服务器将传感器快照和抓拍图片组织成 JSON，调用云端视觉大模型 API 进行多模态异常解释。AI 返回异常类别、场景描述、原因分析和处置建议，显示在 Web Dashboard 的 AI 分析卡片中。需要强调的是，AI 结果仅用于辅助解释和展示，不直接控制水泵、继电器等关键执行器，STM32 本地安全判断始终拥有最高优先级。

### 3:10-3:50 断网/AI失败降级演示

操作：

1. 拔掉 ESP32-CAM 电源或断开 UART。
2. 再次触发烟雾/门磁异常。
3. OLED 显示 CAM_OFFLINE。
4. 蜂鸣器、RGB、风扇/继电器/水泵动作仍正常执行。
5. Web Dashboard 显示"AI 分析不可用——本地判断生效"。

讲解：

> 这是系统的优雅降级能力。摄像头离线后，STM32 标记 CAM_OFFLINE 并在 OLED 提示，但声光报警和执行器保护动作完全不受影响。Web 端的 AI 分析卡片自动切换为"不可用"状态，显示本地判断生效。这体现了"端侧实时控制为核心、云端 AI 为增强"的设计理念。

### 3:50-4:20 MPU6050 设备健康监测

操作：轻敲或搬动设备模型。

现象：OLED 显示 RMS/PP 增大，达到条件后触发振动预警/报警。

讲解：

> MPU6050 使用基线校准、RMS、峰峰值和连续超阈值计数，避免单次碰撞误报。振动数据也作为风险评分输入参与多模态融合判断。

### 4:20-5:00 总结

讲稿：

> 本项目的核心创新在于：第一，多模态感知——6 种传感器数据融合风险评分，不是简单阈值触发；第二，端云协同——本地实时闭环保障安全，云端 AI 提供智能解释增强；第三，优雅降级——摄像头、网络、AI 任一失效都不影响本地安全功能；第四，系统工程——状态机、通信协议、ACK/心跳/重传、错误码、供电安全构成完整工程实践。感谢老师指导，请批评指正。

## 13.7 老师可能提问与回答（AI/Web 强化版）

| 问题 | 回答 |
|---|---|
| 为什么说这是 AI 项目？ | 系统第四层是云端 AI 智能解释层，通过调用视觉大模型 API 对异常图片进行多模态分析，返回异常类别、场景描述和处置建议。AI 不是简单的阈值判断，而是具备图像理解和自然语言解释能力的智能化增强。同时系统的多传感器融合风险评分本身也体现了多模态感知与智能判断思路。 |
| 为什么不用 AI 直接控制水泵？ | 工业安全系统的核心原则是"安全功能不依赖不确定性"。AI 依赖网络、云端服务和图像质量，任何一环失败都会导致安全功能失效。因此关键执行器由 STM32 基于本地传感器确定性判断控制，AI 仅提供辅助解释和归档，确保系统在最差网络条件下仍能保护安全。 |
| 为什么 STM32 仍然是主控？ | 课程要求 STM32F103C8T6 为主控。同时从系统设计角度，安全判断和执行器控制需要确定性和实时性，STM32 裸机调度满足毫秒级响应需求，不依赖网络或操作系统延迟。 |
| 断网时如何运行？ | P0 本地闭环完全不依赖网络。STM32 本地完成传感器采集、风险评分、状态机、OLED 显示、按键交互和执行器控制。断网后 OLED 显示 NET_OFFLINE，Web/MQTT/AI 功能暂停，但蜂鸣器、RGB、风扇、继电器、水泵动作不受影响。 |
| ESP32-CAM 失效时如何运行？ | STM32 每 2 秒发送 PING 心跳。连续 3 次心跳失败后标记 CAM_OFFLINE，OLED 显示摄像头离线。此后所有异常仍由本地处理，只是没有抓拍图片。这就是优雅降级。 |
| 手机端/Web/服务器部分如何体现 A 类特征？ | 系统提供 Flask REST API（8 个端点）、Web Dashboard（5 个功能区）和手机端浏览器查看。服务器保存事件数据、抓拍图片和 AI 分析结果，支持历史事件查询和远程状态监控。这些都超出了 C 类项目"单机传感器阈值报警"的范畴，体现了端云协同的系统工程能力。 |
| 云端 AI 如何参与多模态异常解释？ | 服务器将异常事件的 6 种传感器数据和抓拍图片打包为 JSON，通过 HTTPS 发送给云端视觉大模型。AI 分析图片内容并结合传感器上下文，返回异常类别、场景描述、原因分析、处置建议和置信度。结果在 Web 端以卡片形式展示，同时保存在事件记录中用于归档分析。 |
| 为什么不从零训练模型？ | 两周课程周期不适合从零训练。使用云端 API 调用成熟的视觉大模型，可以快速获得高质量的图像理解和自然语言解释能力，同时降低开发风险。 |
| 如何避免误报？ | 使用滑动均值、连续触发计数、多传感器融合、状态机恢复延时和传感器权重。例如 MQ-2 需要连续 3 次超阈值才触发，火焰需要连续 2 次确认，MPU6050 需要连续多个窗口超阈值才报警。 |
| 水泵安全吗？ | 水泵使用低压供电和驱动板控制，限时运行 1-3 秒，水路有托盘并远离主控和电源。必要时用风扇/LED 替代演示。 |
| 项目最大亮点是什么？ | 多模态感知、本地实时闭环、视觉取证、Web/手机端远程可视化、云端 AI 异常解释、优雅降级、STM32 与 ESP32-CAM 端云协同、设备健康监测和完整系统工程设计。 |

## 13.8 答辩备用策略

| 失败场景 | 备用方案 |
|---|---|
| ESP32-CAM 现场无法抓拍 | 展示本地闭环和已录视频，说明 ESP 是增强模块，降级机制就是亮点 |
| Wi-Fi 不可用 | 不演示 Web/MQTT/AI，只演示 OLED 和串口日志；用预录 Web 截图说明 |
| 云端 AI 超时 | 展示 Web 端"AI 分析不可用"降级界面，说明本地判断生效 |
| 火焰演示受限 | 使用安全光源、按键或手动模式模拟火焰触发 |
| 水泵漏水风险 | 禁用真实喷水，改为空泵、风扇或 LED 模拟 |
| 某个传感器失效 | 用其他传感器触发多级报警，报告中说明降级机制 |
| 面包板接触不良 | 使用热熔胶/扎带固定，准备备用杜邦线，演示手动模式 |
| 现场演示全部失败 | 播放预录完整演示视频和降级演示视频，配合 PPT 和报告讲解 |

## 13.9 答辩加分技巧

### 主动展示降级

不要等老师问"断网怎么办"，主动在演示中拔掉 ESP32-CAM 展示降级，主动说明 AI 超时时的界面表现。

### 使用专业术语

在答辩中自然使用以下术语：多模态融合、端云协同、边缘计算、优雅降级、ACK/心跳/重传、状态机、风险评分、异常分级、视觉取证、REST API、JSON Contract。

### 准备演示视频

至少录制两个视频：正常完整演示视频（3-5 分钟）和降级演示视频（1-2 分钟），保存在手机和 U 盘中。

### 报告和 PPT 配图

Web Dashboard 截图、手机端截图、AI 分析结果截图、串口日志截图务必提前准备。即使现场演示不完美，报告中有完整截图也能说明功能已实现。

### 不要怕 P2 没完成

如果 Web/AI 没有全部完成，答辩重点放在：

1. STM32 主控边界和本地闭环（P0）。
2. 状态机和风险评分设计（P0）。
3. ESP32-CAM 抓拍或离线降级（P1）。
4. 供电安全和工程可靠性。
5. 系统工程思维和完整文档。

然后说明 Web/AI 已有设计方案和 JSON 协议定义，因时间原因部分功能尚在调试中，但架构已就绪。
