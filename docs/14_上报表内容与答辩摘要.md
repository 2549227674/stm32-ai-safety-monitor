## 14.3 上报表基本功能：完整版

本系统面向实验室设备柜、储物柜、机房设备箱等场景，以 STM32F103C8T6 为唯一实时安全主控，融合 MQ-2 烟雾、火焰、PIR 人体红外、门磁、温湿度和 MPU6050 振动等多模态传感器，构建本地风险评分引擎与多级状态机。系统通过 OLED 实时显示传感器参数、风险等级和故障状态，利用 4 位按键完成布防、撤防、静音、复位等交互操作；异常发生时，STM32 驱动蜂鸣器分级报警、RGB LED 状态指示、风扇通风、继电器断电保护和水泵喷淋模拟等执行器完成本地保护动作。

系统通过 UART 主从协议调度 ESP32-CAM 边缘视觉协处理器，实现异常现场抓拍取证，协议支持 ACK 确认、PING/PONG 心跳、超时重传和错误码；抓拍图片与传感器快照通过 Wi-Fi 上传至 Flask 轻量服务器，Web Dashboard 和手机端浏览器可远程查看当前状态、历史事件和抓拍图片。服务器将传感器快照与抓拍图片组织为结构化 JSON，调用云端视觉大模型 API 进行异常类别识别、场景描述生成和处置建议输出，AI 分析结果在 Web 端以卡片形式展示。

ESP32-CAM 离线、网络失败或 AI 超时时，系统自动标记故障并降级，STM32 本地安全闭环完全不受影响。整个系统形成"端侧实时控制 → 边缘视觉取证 → 远程可视化管理 → 云端智能分析"四层端云协同 AI 安全巡检架构。

## 14.14 为什么这是 AI 项目

答辩中老师可能质疑"AI 在哪里"，以下是防守要点：

| 质疑点 | 回应 |
|---|---|
| "AI 只是调了个 API" | 系统完成了从传感器快照采集、图片上传到 AI 请求构建、响应解析、结果展示的完整端云 AI 链路；同时多传感器融合风险评分本身就体现了多模态智能判断思路 |
| "本地没有 AI" | 本项目采用端云协同架构，端侧负责实时安全控制，云端负责智能化解释增强，这是工业 AIoT 的标准设计范式 |
| "不是自己训练的模型" | 两周课程周期不适合从零训练。使用云端大模型 API 是工程最优解，重点在于如何构建请求上下文、解析结构化响应、处理降级和将结果集成到 Web 展示 |
| "AI 不参与控制" | 安全系统的核心原则是确定性和实时性。AI 依赖网络和云端服务，不适合作为安全动作的前置条件。AI 用于事后复核和辅助分析是正确的系统设计决策 |

## 14.15 "AI 主线但本地可靠"答辩话术

### 讲系统定位时

> 我们的系统是端云协同 AI 安全巡检系统。AI 是系统架构的顶层设计目标，不是可有可无的附加功能。整个四层架构的设计目标就是把异常事件的感知、取证、传输和智能分析形成完整链路。

### 讲安全设计时

> AI 之所以不直接控制执行器，是因为工业安全系统要求关键安全功能不依赖不确定性。网络延迟、AI 超时、云端故障都是不确定因素。我们的设计是：STM32 本地确定性控制保障安全底线，云端 AI 提供智能化增值解释，两者协同但职责分明。

### 讲降级时

> 当 AI 不可用时，Web 端自动显示"AI 分析不可用——本地判断生效"。这不是系统的缺陷，而是优雅降级设计的体现。真正好的 AI 系统不是"AI 挂了全挂了"，而是"AI 挂了照样安全"。

### 讲创新点时

> 项目的核心创新不是"用了 AI"，而是"在嵌入式安全系统中正确地引入了 AI"。我们解决的是：如何在网络不可靠、算力受限、安全要求高的场景下，让 AI 发挥辅助解释和智能分析的价值，同时确保系统在最差条件下仍然安全可用。

## 14.16 为什么符合 STM32F103C8T6 主控要求

| 要求 | 本项目实现 |
|---|---|
| 本地传感器采集 | MQ-2、火焰、PIR/门磁、温湿度、MPU6050 由 STM32 采集 |
| 状态机 | INIT → IDLE → ARM_DELAY → NORMAL → PRE_ALARM → ALARM → DANGER → SILENCE → RECOVER → FAULT → MANUAL 由 STM32 管理 |
| OLED 显示 | OLED 由 STM32 直接 I2C 驱动 |
| 按键交互 | 4 位按键由 STM32 扫描、消抖和处理 |
| 蜂鸣器/LED | 蜂鸣器和 RGB LED 由 STM32 GPIO/PWM 控制 |
| 继电器/水泵/风扇 | 关键执行器由 STM32 通过驱动模块控制，ESP32-CAM 不参与 |
| 异常分级 | 多传感器融合风险评分和等级判断由 STM32 完成 |
| 本地保底 | 断开 ESP32-CAM、关闭网络、AI 不可用时仍可完整演示 P0 |
| 外部模块通信 | STM32 通过 UART 主从协议调度 ESP32-CAM |
| 日志与可观测性 | 串口日志、事件计数、OLED 故障标志由 STM32 管理 |

## 14.17 为什么不是普通烟雾报警器

普通烟雾报警器："MQ-2 超阈值 → 蜂鸣器响"。本项目在此基础上增加了：

1. **多模态感知**：不只看烟雾，六类传感器数据融合判断。
2. **风险评分**：不只一条阈值，加权评分分级响应。
3. **多级状态机**：不只响/不响，正常→预警→报警→危险→静音→恢复→故障清晰分级。
4. **执行器联动**：不只蜂鸣器，风扇通风、继电器断电保护、水泵喷淋模拟。
5. **OLED/按键/RGB 交互**：不只听到响声，看到状态、操作系统、获得反馈。
6. **边缘视觉取证**：异常时 ESP32-CAM 自动抓拍，保留现场证据。
7. **Web/手机端查看**：不只现场知道报警，远程也能看到状态和图片。
8. **云端 AI 解释**：不只知道"报警了"，还知道"AI 认为是什么异常、原因是什么、建议怎么处理"。
9. **优雅降级**：不是"摄像头坏了全完了"，而是摄像头/网络/AI 任一失效本地照常工作。
10. **完整系统工程**：通信协议、ACK/心跳/重传、错误码、供电安全、设备健康监测。

## 14.18 如果第三四层没完成，如何不影响答辩

> 本项目的安全闭环由 STM32 本地完成。第一层已经具备多传感器采集、风险评分、状态机、OLED 显示、按键交互、声光报警和执行器联动。ESP32-CAM 抓拍属于第二层增强，Web/服务器/手机端属于第三层扩展，云端 AI 属于第四层智能化提升。即使第三四层尚在调试中，也不影响系统的核心安全监测功能与现场演示。

答辩重点应放在：

1. STM32 主控边界和第一层完整闭环演示。
2. 状态机和风险评分设计——这体现系统工程能力。
3. ESP32-CAM 抓拍或离线降级演示——这体现端云协同和可靠性。
4. 供电安全和工程可靠性——这体现工程素养。
5. 四层架构设计文档——即使代码未全部完成，架构设计、JSON Contract、API 定义已经完整，说明团队具备系统设计能力。

关键话术：

> 我们的架构设计是完整的，JSON 协议、AI 请求响应格式、REST API 都已定义并写入文档。第三四层的代码框架已就绪，部分功能因调试时间原因尚未在现场稳定演示，但架构能力和工程思路已经体现在设计文档和测试报告中。

## 14.20 一句话项目定位（可直接粘贴）

### 学术版（报告摘要/上报表用）

> 本系统以 STM32F103C8T6 为实时安全主控，融合六类多模态传感数据构建本地风险评分与状态机控制闭环，通过 ESP32-CAM 边缘视觉协处理实现异常抓拍取证，并经 Web 服务器上传至云端 AI 进行图像复核与异常解释，形成端侧控制、边缘取证、远程管理、云端智能四层协同的 AI 安全巡检系统。

### 口语版（答辩/项目介绍用）

> 一句话说，我们做了一个基于 STM32 的智能安全巡检盒：本地六种传感器检测到异常后，STM32 自动报警保护；同时拍照上传到手机端查看，云端 AI 还会告诉你"这是什么异常、为什么报警、建议怎么做"。摄像头坏了、网断了、AI 挂了，本地照样能报警。

### 简历版

> 设计并实现端云协同 AI 多模态安全巡检系统：STM32 主控完成六类传感器融合风险评分与状态机联动控制，ESP32-CAM 异常抓拍取证，Flask 服务器提供 Web Dashboard 和 REST API，云端 AI 实现异常图像解释与处置建议生成，系统支持全链路优雅降级。
