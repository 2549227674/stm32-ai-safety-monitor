## 13.2 架构迁移说明写法

推荐表述：

```text
第一版采用 STM32 + ESP32-CAM + Flask + 云端 AI 的端云协同架构。实际联调中，
ESP32-CAM GPIO13 物理桥接尚未完成最终验收。考虑到课程周期、调试效率和本地 AI
目标，第二版迁移为 i.MX6ULL + Orange Pi 5 的端边协同架构。

末期 i.MX6ULL 出现供电/启动异常后，团队复用已完成的 Linux 安全主控逻辑，将其
迁移到 Orange Pi 5 上作为独立特权进程。AI 推理服务仍保持独立进程，仍不直接
控制执行器；本地安全闭环在断网、断 AI 时仍可依据传感器进入安全态。
```

不要写"STM32 失败"；写"第一版暴露工程调试成本，第二版进行平台升级"。
不要写"i.MX 坏了所以放弃"；写"平台迁移与工程止损"。

## 13.3 当前主线架构

```text
OPi5 safetyd + OPi5 AI + device-agent + Flask/React
```

| 模块 | 进程 | 职责 |
|---|---|---|
| opi5-controller | `opi5_safetyd` | GPIO 传感器 → 状态机 → 执行器 |
| opi5-ai | `opi5-ai-qwen3vl.service` | Qwen3-VL 推理，`control_allowed=false` |
| opi5-device-agent | `opi5-device-agent.service` | 心跳/遥测/视频/AI observation |
| Flask backend | `app.py` | 事件/遥测/AI/通知 API |
| React edge-console | Vite build | Dashboard 可视化 |

## 13.4 硬件系统章节

必须包含：

- OPi5 26pin 引脚分配与接线图
- 传感器：PIR/火焰/MQ-2
- 执行器：buzzer/RGB/Water MOS/Pan 舵机
- I2C：OLED + MPU6050
- 摄像头：USB 摄像头
- 供电拓扑与星形共地
- 安全边界：低压、不接 220V、AI 不控执行器

## 13.5 软件系统章节

必须包含：

- `opi5_safetyd` 模块图
- `opi5_ai_service` 接口（`/health`、`/api/infer/vision`）
- `opi5-device-agent` 功能（心跳/遥测/视频/AI observation）
- Flask API 总览
- React edge-console 页面
- systemd 三进程独立管理
- 历史 i.MX 阶段作为工程迭代证据

## 13.9 失败降级演示

1. **停止 `opi5-ai-qwen3vl`**：`opi5_safetyd` 仍能依据本地传感器进入 ALARM。
2. **断开 Flask**：device-agent 事件进入 spool，恢复后补发。
3. **摄像头不可用**：device-agent 回退 mock 模式，视频标注 MOCK。

## 13.10 答辩问答准备

| 问题 | 回答要点 |
|---|---|
| 为什么不用 STM32 了？ | 课程周期内 Linux + SSH 更利于调试；保留底层 GPIO/I2C/PWM 证据 |
| 为什么从 i.MX 迁到 OPi5？ | i.MX6ULL 末期供电/启动异常，复用安全主控逻辑迁移到 OPi5，体现工程止损 |
| 一块 OPi5 同时跑控制和 AI？ | 两进程独立，safetyd 轻量 GPIO 轮询 + 状态机，AI 推理互不依赖 |
| Linux 不是硬实时怎么办？ | 人机尺度联动，目标 100ms–500ms；默认安全态、watchdog、离线降级 |
| AI 会不会误控？ | AI 不直接控制执行器，`control_allowed=false` 硬编码 |
| 这还是数字系统吗？ | 有真实 GPIO/PWM/I2C、嵌入式 Linux 部署，体现数字系统综合训练 |
| 为什么迁移是加分？ | 三次迭代体现完整工程过程和平台迁移能力 |
