## 13.2 架构迁移说明写法

推荐表述：

```text
第一版采用 STM32 + ESP32-CAM + Flask + 云端 AI 的端云协同架构。实际联调中，网络侧和 STM32 UART 输出侧已取得阶段性验证，但 ESP32-CAM GPIO13 物理桥接尚未完成最终验收。考虑到课程周期、调试效率和本地 AI 目标，第二版迁移为 i.MX6ULL + Orange Pi 5 的端边协同架构。
```

不要写”STM32 失败”；写”第一版暴露工程调试成本，第二版进行平台升级”。

### OPi5 临时主控迁移口径

答辩时如被问到”为什么只用一块板”，推荐表述：

```text
末期 i.MX6ULL 出现供电/启动异常后，团队没有推倒重来，而是复用已完成的 Linux 安全主控
逻辑，将其迁移到 Orange Pi 5 上作为独立特权进程。AI 推理服务仍保持独立进程，仍不直接
控制执行器；本地安全闭环在断网、断 AI 时仍可依据传感器进入安全态。这体现了工程上的
止损能力和架构弹性。
```

不要写”i.MX 坏了所以放弃”；写”平台迁移与工程止损”。

## 13.6b 全无线演示口径

本项目最终演示优先采用全无线热点链路。PC 和 OPi5 均连接同一手机热点；OPi5 通过 USB WiFi 直接访问 PC Flask Dashboard。

演示流程：

1. 打开手机热点。
2. PC 连接热点并启动 Flask Dashboard（`python3 app.py --host 0.0.0.0`）。
3. OPi5 上电后自动连接热点。
4. PC 检查 OPi5 WiFi IP。
5. Dashboard 打开。
6. 执行本地告警 / AI infer / Dashboard event。
7. 故障时插回网线使用回退链路。

全无线只是网络承载方式变化；当前本地安全闭环由 OPi5 上的 `opi5_safetyd` 完成，AI 推理由同板独立 `opi5_ai_service` 完成。两进程通过本机 HTTP/文件/事件接口解耦；AI 不直接控制执行器；无网络、无 Flask、无 AI 时，本地传感器仍可触发蜂鸣器/RGB/MOS 等安全动作。

回退方案：

- 全有线：OPi5 → PC 直连。
- 答辩现场保留网线。

## 13.8 失败降级演示

建议演示三个场景：

1. **停止 `opi5_ai_service`**：`opi5_safetyd` 仍能依据本地传感器进入 ALARM，上报 `ai_status=offline`，不崩溃。
2. **断开 Flask**：`opi5_safetyd` 事件进入本地 spool，恢复后可 flush 补发。
3. **PCA9685 不可用**：不影响 GPIO/PWM 主线，PCA9685 当前已降级为可选后端。

## 13.9 答辩问答准备

| 问题 | 回答要点 |
|---|---|
| 为什么不用 STM32 了？ | 课程周期内 Linux + SSH 更利于调试；保留底层 GPIO/I2C/PWM 证据 |
| 为什么从 i.MX 迁到 OPi5？ | i.MX6ULL 末期供电/启动异常，团队复用已完成的 Linux 安全主控逻辑迁移到 OPi5，体现工程止损和架构弹性 |
| 一块 OPi5 同时跑控制和 AI，会不会互相影响？ | 两进程独立，`opi5_safetyd` 是轻量级 GPIO 轮询 + 状态机，`opi5_ai_service` 是推理服务，互不依赖；AI 崩溃不影响安全闭环 |
| Linux 不是硬实时怎么办？ | 人机尺度联动，目标 100ms–500ms；默认安全态、watchdog、离线降级 |
| AI 会不会误控？ | AI 不直接控制执行器，只返回 `risk_hint`；`control_allowed=false` 硬编码 |
| PCA9685 坏了为什么还能演示？ | PCA9685 确认为模块问题（4 个均未 ACK），当前主线改用 GPIO/PWM 直控，PCA9685 保留为后续替换模块后的可选后端 |
| 这还是数字系统吗？ | 有真实 GPIO/PWM/MOS、I2C OLED/MPU6050、嵌入式 Linux 部署，体现数字系统综合训练的核心能力 |
| 为什么迁移是加分？ | 展示工程问题定位、平台选择和架构重构能力；三次迭代（STM32→i.MX→OPi5）体现完整工程过程 |
