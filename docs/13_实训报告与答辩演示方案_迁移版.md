## 13.2 架构迁移说明写法

推荐表述：

```text
第一版采用 STM32 + ESP32-CAM + Flask + 云端 AI 的端云协同架构。实际联调中，网络侧和 STM32 UART 输出侧已取得阶段性验证，但 ESP32-CAM GPIO13 物理桥接尚未完成最终验收。考虑到课程周期、调试效率和本地 AI 目标，第二版迁移为 i.MX6ULL + Orange Pi 5 的端边协同架构。
```

不要写“STM32 失败”；写“第一版暴露工程调试成本，第二版进行平台升级”。

## 13.6b 网络优化演示口径

现场演示时可减少 OPi5 网线：

- i.MX 通过 Windows portproxy 访问 OPi5 WiFi AI 服务。
- 拓扑：i.MX → PC:18080 → OPi5 WiFi:8080。
- 如 WiFi 不稳定，插回 OPi5 网线恢复全有线：i.MX → OPi5 10.0.1.120:8080。
- 演示前检查：`curl -sS --connect-timeout 5 http://192.168.137.1:18080/health`（portproxy）或 `http://10.0.1.120:8080/health`（全有线）。

## 13.9 答辩问答准备

| 问题 | 回答要点 |
|---|---|
| 为什么不用 STM32 了？ | 课程周期内 Linux + SSH 更利于调试；保留底层 GPIO/I2C/PWM 证据 |
| Linux 不是硬实时怎么办？ | 人机尺度联动，目标 100ms–500ms；默认安全态、watchdog、离线降级 |
| AI 会不会误控？ | AI 不直接控制执行器，只返回 `risk_hint` |
| 这还是数字系统吗？ | 有真实 GPIO/I2C/PWM/MOS、逻辑分析仪、嵌入式 Linux 部署 |
| 为什么迁移是加分？ | 展示工程问题定位、平台选择和架构重构能力 |
